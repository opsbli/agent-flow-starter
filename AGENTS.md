# agent-flow-starter Agent 规则

> 本仓库是 `agent-flow` AI 开发流程的通用 starter。
> 保持通用，不要写入具体业务项目的历史、需求或领域规则。

## 默认取向

除非用户明确覆盖，否则以下规则适用于本项目中的所有任务。

**默认取向**：非琐碎任务中，谨慎优先于速度；简单任务可以自行判断。

### 规则 1 —— 编码前先思考

明确说明假设。如果不确定，先问，不要猜。存在多种解释时，先列出解释，不要静默选择。如果有更简单的方案，要提出异议。困惑时停止，说明哪里不清楚。

### 规则 2 —— 简洁优先

使用能解决问题的最少代码。不要写投机性功能。不要添加需求之外的功能。不要为一次性代码做抽象。检验标准：资深工程师会觉得它过度复杂吗？如果会，就简化。

### 规则 3 —— 外科手术式修改

只改必须改的地方。只清理自己造成的问题。不要"顺手改进"相邻代码、注释或格式。不要重构没坏的东西。匹配现有风格。每一行改动都应该能直接追溯到用户请求。

### 规则 4 —— 目标驱动执行

定义成功标准，循环验证直到达成。把模糊任务转化成可验证目标。强成功标准能让你独立迭代。

### 规则 5 —— 只把模型用于需要判断力的地方

把模型用于分类、起草、总结、从非结构化文本中提取信息。不要把模型用于路由、重试、状态码处理或确定性转换。如果代码能回答，就让代码回答。

### 规则 6 —— Token 预算不是建议

单任务预算：4,000 token。单 session 预算：30,000 token。接近预算时，先总结，再重新开始。暴露预算突破，不要静默超支。

### 规则 7 —— 暴露冲突，不要折中平均

如果代码库中的两个既有模式互相矛盾，不要混合它们。选择其中一个，优先选择更新或测试更多的模式。解释选择原因，并把另一个模式标记为后续清理项。

### 规则 8 —— 写之前先读

添加代码前，先阅读当前文件的 exports、直接调用方，以及明显相关的共享工具。如果不理解现有代码为什么这样组织，先问，再添加。"看起来和我无关"是危险信号。

### 规则 9 —— 测试要验证意图，而不只是行为

测试必须表达行为为什么重要，而不仅是它做了什么。如果业务逻辑变化时测试不会失败，这个测试就是弱测试或错误测试。不要把浅层测试通过当成正确性的证明。

### 规则 10 —— 每个重要步骤后设置检查点

每完成一个重要步骤，总结已完成内容、已验证内容、剩余内容。如果无法向用户描述当前状态，就不要继续。如果丢失上下文，停止并重新陈述。

### 规则 11 —— 匹配代码库约定，即使你不同意

在代码库内部，遵守约定优先于个人品味。使用现有命名、结构、测试风格和错误处理模式。如果某个约定确实有害，要显式提出，不要静默分叉出另一套风格。

### 规则 12 —— 大声失败

如果有任何内容被静默跳过，就不能说"已完成"。如果跳过了相关测试，就不能说"测试通过"。默认暴露不确定性，而不是隐藏它。

## 默认入口

在本 starter 仓库内工作时，先读：

```text
agent-flow/README.md
```

修改流程本身时，还必须读：

```text
agent-flow/GO.md
agent-flow/manifest.yaml
agent-flow/core/source-of-truth.md
agent-flow/core/evolution.md
```

## 编辑规则

- `AGENTS.md` 只保留短规则和硬约束。
- 详细用法写入 `README.md` 或 `agent-flow/README.md`。
- 模板必须保持通用，不写具体项目名称、模块名、业务状态或历史结论。
- 不要把真实项目的 `changes/`、`logs/`、`reports/`、`known-good-baselines/` 放进 starter。
- 修改脚本时，凡是适用的能力必须同时更新 Windows `.ps1` 和 Linux/macOS `.sh`。
- 修改脚手架结构后，必须运行两套 `scaffold-health`。

## ECC 能力集成（可选）

如果本机已安装 ECC（`pi install npm:ecc-universal`），在执行 agent-flow 流程时可按需调用 ECC 技能加速。

映射表见 `agent-flow/ecc-integration.md`。关键节点：

| agent-flow 步骤 | ECC 加速能力 | 调用方式 |
|----------------|-------------|---------|
| CODE_SCAN 代码扫描 | ecc-explorer agent | `@ecc-explorer 扫描项目结构和相似模块` |
| DESIGN 架构设计 | ecc-architect agent | `@ecc-architect 设计方案架构` |
| 验证阶段 | /ecc-review + /ecc-quality | 一键代码审查和质量门禁 |
| 安全审计 | /ecc-security | 扫描密钥和 OWASP 漏洞 |
| 实现阶段 | 各语言 skill（自动匹配） | AI 自动加载对应语言模式 |

pi 中还有便捷命令：`/af-scan`、`/af-design`、`/af-verify`、`/af-go` 可快速执行对应步骤。

## 安装契约

starter 必须支持：

- 在目标项目创建或更新 `agent-flow/`。
- 在目标项目创建或更新 `AGENTS.md` 中的 `agent-flow` 区块。
- 在目标项目创建或更新 `agent-flow/ecc-integration.md`（可选集成）。
- 除非显式强制覆盖，否则保留目标项目的 `agent-flow/changes`、`agent-flow/logs`、`agent-flow/reports`。
