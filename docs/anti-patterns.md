# Anti-Patterns: When NOT to use agent-flow

> 明确 agent-flow 的边界——哪些场景不适合 agent-flow，以及如何选择更轻量的方案。

---

## 什么情况下 agent-flow **不适合**你的项目

### 1. 纯原型/一次性脚本

**特征**：
- 代码写完后会被丢弃
- 不需要版本管理
- 不需要审计

**为什么不适合**：agent-flow 的门禁、工件、验证体系是为**持续维护的代码**设计的。一次性脚本走完整流程的成本超过收益。

**替代方案**：直接写代码。如果确实需要记录，用一个简化版的 `CHANGE.md` + `VERIFY.md` 即可。

---

### 2. 个人项目 / 单人开发且无外部审计要求

**特征**：
- 没有合作者
- 没有合规/审计要求
- 代码出了问题是"自己的事"

**为什么不适合**：agent-flow 的核心价值在于**多人协作时的可追溯和可控性**。单人项目可以享受其中的好习惯（Code-First Scan、TDD），但不需要全流程。

**替代方案**：跳过 DESIGN.md、PLAN.md、AUDIT.md，只使用 Light 流程。

---

### 3. 变更只有一行代码或纯配置修改

**特征**：
- 修改一个 YAML 文件的某个值
- 调整 CSS 颜色
- 改一个日志级别

**为什么不适合**：创建 Change 工件的时间可能比改代码本身还长。

**替代方案**：直接提交。如果确实想记录，写一行 commit message 就够了。

> ⚠️ **注意**：如果这个"一行修改"涉及权限、数据库、生产配置，仍然建议走 Light 流程，因为这些修改的**风险不取决于代码行数**。

---

### 4. 项目尚未确定技术栈或架构方向

**特征**：
- 项目还在探索期
- API 接口频繁变化
- 数据库 schema 每天在改
- 团队还没定编码规范

**为什么不适合**：agent-flow 的 Design Alignment、Plan Audit、Closure Audit 等门禁假设项目已经有一个**相对稳定的基线和架构约定**。在探索期，这些门禁会成为僵化的障碍。

**替代方案**：项目稳定后再引入 agent-flow。`init-project.ps1` 可以在任何阶段运行。

---

### 5. 团队没有代码审查文化，且拒绝接受

**特征**：
- "写完就上线"是常态
- 代码审查被视为"浪费时间"
- 没有质量门禁的接受度

**为什么不适合**：agent-flow 本质上是一个**流程约束工具**。如果团队不接受流程约束，agent-flow 只会被绕开或忽略，变成无人维护的"文档废物"。

**替代方案**：先在团队中建立代码审查习惯，再引入 agent-flow。可以从 Light 流程开始，逐步增加门禁。

---

### 6. 单一文件改动且不涉及任何业务逻辑

**特征**：
- 只改了 README.md 的拼写错误
- 更新了依赖版本（无 API 变化）
- 添加了注释

**为什么不适合**：这些改动的风险为零或极低，不需要门禁验证。

**替代方案**：直接提交。

---

## 何时应该降级流程（而不是升级）

agent-flow 的流程设计是**风险驱动的**。如果以下情况发生，应该考虑**降级**而不是升级：

| 当前级别 | 降级条件 | 降级到 |
|---------|---------|--------|
| Heavy | 评审发现实际变更量很小，且不涉及 schema/auth/public API | Standard |
| Standard | 变更经过 Code Scan 确认只有一处修改，且不涉及权限 | Light |
| Light | 变更被确认只有注释/配置/非功能性修改 | 直接提交 |

**降级操作**：在 `CHANGE.md` 中修改 Flow Level 标记，重新运行对应流程。

---

## 何时应该跳过某个门禁（而不是全部绕过）

| 门禁 | 可跳过的条件 | 必须记录的原因 |
|------|-------------|--------------|
| alignment-check | Light 流程不需要 | — |
| plan-check | Standard 流程不需要 | — |
| code-drift-check | Change 不涉及代码文件（纯文档/配置） | 在 VERIFY.md 中记录 |
| evolution-check | Light 流程不需要（但推荐写） | 在 CHANGE.md 中注明 |
| closure-check | Emergency 通道 | 在 AUDIT.md 中记录回填计划 |

---

## 健康使用检查清单

如果你的项目出现以下**三个或以上**症状，建议重新评估 agent-flow 的使用方式：

- [ ] 团队成员开始"忘记"运行门禁
- [ ] CHANGE.md 的内容越来越敷衍
- [ ] EVOLUTION.md 持续写"本次无改进建议"
- [ ] 提交信息不再引用 AC 编号
- [ ] CI 中的门禁经常被跳过或忽略
- [ ] 新成员入职时觉得流程"看不懂"
- [ ] 迭代速度明显下降（不是因为代码复杂性增加）

**诊断建议**：

1. 检查是否有太多 Heavy 流程的 Change → 考虑将其降级
2. 检查是否门禁过多导致疲劳 → 运行 `gate-fatigue-check` 分析
3. 检查是否需要重新培训 → 运行 `af-quickstart` 重新走一遍
